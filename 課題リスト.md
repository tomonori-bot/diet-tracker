# BodyLog 課題リスト

> 本格的に作り直す／本番運用するときに対応する項目をまとめたもの。
> 最終更新：2026-06-04

---

## 🔴 最優先（これを直さないと実運用できない）

### 1. データが消える（永続化されていない）
**症状**
- 新規登録してもしばらくするとログインできなくなる
- 毎回ログインを求められる
- 登録した顧客データもいつの間にか消える

**原因**
- 全データ（アカウント・ログイン状態・顧客情報）を `data/db.json` という1ファイルに保存している
- Renderの**無料プランには永続ディスクがない**ため、
  - 約15分アクセスが無いとサーバーが眠り、次回まっさらで再起動
  - 再デプロイのたびにもリセット
  - そのたびに `db.json` が消える

**解決方針（長期的におすすめ＝外部DB）**
- [ ] **外部DB（Supabase など無料クラウドDB）に移行する** ← 推奨。サーバーを増やしても引っ越しても データが残る正攻法
  - 作業：`db.json`のファイル読み書きを、DBへの読み書きに置き換える
  - Supabaseの無料アカウント作成が必要
- [ ] （簡易策）Renderの有料永続ディスク（月7ドル〜）を使い、`db.json`をそこに置く
  - メリット：コード変更最小。デメリット：月額発生・将来スケールしにくい

> ※ 作り直すタイミングで、最初からDB前提で設計するのが効率的。

## 🟡 中優先（運用前に確認・改善したい）

### 2. パスワードを忘れたときの復旧手段がない
- 今はパスワードを忘れるとそのアカウントに入れなくなる
- [ ] パスワードリセット機能（メール送信など）を検討

### 3. 「30日間ログイン保持」チェックが効いていない可能性
- チェックボックスはあるが、サーバーが頻繁に再起動するため保持されない
- 課題1（永続化）を直せば改善する見込み。直後に再確認する
- [ ] 永続化後に、ログイン保持が実際に効くか検証

### 4. 写真の保存先も消える
- アップロードした写真は `data/photos/` に保存しているが、これも課題1と同じ理由で消える
- [ ] 写真は外部ストレージ（Supabase Storage / Cloudinary など）に保存する設計に

### 5. 誰でも登録できてしまう
- 現状、URLを知っていれば誰でもアカウントを作れる
- 本番では招待コード制やメール認証を検討
- [ ] 登録に制限をかけるか方針決定

### 6. 運営者用の管理画面（全トレーナーを管理する上位画面）
- サービスとして運営するなら、運営者（自分）が全トレーナーを把握・管理する画面が必要
- 現状は全員が同じ `role: 'admin'` で、トレーナー同士は完全分離。上から束ねる視点がない
- 作るとできること（例）：
  - 登録トレーナー一覧（誰が・いつ登録・顧客何人・最終ログイン・アクティブか）
  - 全体の数字（総トレーナー数・総顧客数・成長推移）
  - 問題アカウントの停止・削除
- **前提**：課題1（永続化）を先に解決すべき。土台が消えるDBの上に管理画面を建てても、再起動でトレーナーごと消えるため
- 設計判断が必要：何を管理するか／課金や権限（role: owner と admin の分離）をどうするか
- [ ] 課題1の解決後に、設計を相談してから着手

## 🟢 低優先（あると良い・将来の拡張）

### （検討中）患者画面を朝用・夜用に分ける
- 体重は朝イチ、トレーニング・振り返りは夜、と記録タイミングが違う
- 案A（同じ日に何度でも追記送信できる）は実装済み。まずこれで運用する
- さらに使い勝手を上げるなら「朝の記録／夜の記録」をタブやボタンで分ける案Bがある
- メリット：いつ何を入れるか明確、朝は体重だけのスッキリ画面
- デメリット：UIが複雑化、生活が不規則な人には逆に合わない、作りが重い
- [ ] 実際の顧客の使われ方を見て、案Aで足りなければ案Bを検討

### 6. 実画面のスクリーンショット入り営業資料
- 今の使い方ガイドは図解。実際の画面を入れるとより説得力が出る
- [ ] テスト顧客を入れて画面キャプチャ → 資料に反映

### 7. 顧客への通知（リマインド）
- 記録を忘れている顧客に「今日の記録どうですか?」を送る仕組み
- [ ] LINE通知やプッシュ通知を検討

### 8. データのエクスポート
- 顧客データをCSV等で書き出せると、引き継ぎ・分析に便利
- [ ] エクスポート機能
