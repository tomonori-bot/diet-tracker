# 開発の経緯まとめ（時系列）

## 出発点
- 元は「ダイエット進捗トラッカー」という別バージョンが Render で稼働していた。
- ローカルの webapp 版と、GitHubの diet-tracker 版が枝分かれしていた。
- 方針：本番(diet-tracker)を土台に、欲しい機能を統合していく形に決定。
- → URLやログイン機能を保ったまま、本番リポジトリ(tomonori-bot/diet-tracker)で開発を継続。

---

## 2026-06-01　機能の土台づくり
- 写真アップロード／セルフトレーニングのチェックリスト／継続率の可視化を追加。痛み・姿勢スコアは削除。
- **複数アカウント対応**：新規登録機能、アカウントごとに顧客データを分離。
- **問診票**（事前カウンセリングアンケート）：URL/QRで送信→記入→自動で顧客登録。
- **セッションの種目記録**：種目×重量×回数×セット。マイ種目リスト方式（トレーナーが自分の種目を育てる）。
- **デザイン刷新**：患者画面=エメラルド×丸ゴシック（親しみ）／管理画面=Clear Skyウォーム紙×明朝（上品）。
- **絵文字の削減**：管理画面を特にミニマルに（機能記号だけ残す）。

## 2026-06-09〜10　データ永続化（最大の山）
- 課題：Render無料プランは再起動でデータが消える。本番運用の前提として致命的。
- DB選定：AWS/Firebase/Supabaseを比較し、**Supabase**に決定（今のコードと相性◎、写真も将来対応可）。
- 設計：**ハイブリッドB**（顧客=1行JSON、差分保存）。長期の楽さと締切の両立。
- 実装：`db.js` を新設。当初supabase-jsを使ったが Node20でWebSocketエラー → **REST直叩き**に変更（依存ゼロ）。
- 移行中に踏んだバグ（すべて本番前に修正）：
  - uuid(v14 ESM)がrequireと非互換 → crypto.randomUUID()に置換
  - WebSocket依存 → REST化
  - 環境変数に★混入 → 自動クリーニング追加
  - URL二重結合(/rest/v1/rest/v1) → ドメインだけ取り出す正規化
  - select/deleteのカンマをencodeして404 → 修正
- ローカル(Supabaseモード)で検証 → 本番デプロイ → **本番でSupabase保存・読み出しを確認**。
- **データ永続化、完了。** Supabaseのpatientsテーブルに実データ保存を確認済み。

## 技術メモ（次の人向け・ハマりどころ）
- **Push**はGitHub Desktopで（コマンドラインのpushは認証が通らない）。
- **Supabaseの管理画面**は翻訳拡張で壊れることがある→シークレットウィンドウ推奨。
- **環境変数**（URL/KEY）は値だけを設定。★や囲み記号を一緒にコピーしない（コードで掃除はするが）。
- **service_roleキー**を使う（anonキーではない）。eyJ で始まる長い文字列。
- ローカル検証用に /tmp に Node v20 を展開して使った（この環境にNodeが標準で無いため）。
